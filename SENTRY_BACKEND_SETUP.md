# Sentry Backend Setup

This document is the operator reference for the backend Sentry integration
(`@sentry/node` v10). Frontend Sentry has its own document
(`OBSERVABILITY_SETUP.md §4`); this one is server-side only.

## 1. Initialization order

Sentry's Node SDK auto-instruments Express, HTTP, fs, child_process, and the
OpenTelemetry pipeline by patching modules at `require` time. If
`Sentry.init()` runs **after** the modules it should patch are loaded, those
spans are lost. We solve this with a sidecar `backend/src/instrument.ts`
that is the very first import of every entrypoint:

```ts
// backend/src/server.ts
import "./instrument";   // ← MUST be first
import { createServer } from "http";
import app from "./app";
// ...
```

Same pattern in `backend/src/worker.ts`.

## 2. Configuration (`backend/src/lib/sentry.ts::initSentry`)

| Option | Value | Why |
|---|---|---|
| `dsn` | `process.env.SENTRY_DSN` | empty → SDK is a no-op (safe to deploy without provisioning) |
| `environment` | `process.env.NODE_ENV` | discriminates dev / staging / production |
| `release` | `RENDER_GIT_COMMIT[:7]` | matches the source-map upload tag |
| `tracesSampleRate` | `0.1` prod / `1.0` dev | per the Sentry skill |
| `profilesSampleRate` | `0.1` prod / `0` dev | optional CPU profiling |
| `beforeSend` | strips `cookies` + `Authorization` + `Cookie` headers; drops `/health` URLs; injects `correlation_id` from `AsyncLocalStorage` | cross-references Sentry events with Pino logs |

## 3. Express error handler

`backend/src/app.ts` registers Sentry's error middleware AFTER all routes
and BEFORE our custom error handler:

```ts
Sentry.setupExpressErrorHandler(app);

app.use((err, req, res, _next) => {
  // localized Arabic response shaping
});
```

Sentry's middleware captures the error event with the full request context
(headers, query, route pattern, status code), then calls `next(err)` so our
custom handler still produces the Arabic-text user-facing response.

## 4. Global handlers

Sentry v10 enables `onUncaughtExceptionIntegration` and
`onUnhandledRejectionIntegration` by default. We additionally install
**flush-on-exit** handlers in `instrument.ts` because the default
integration captures the error but the process can still exit before the
network event finishes:

```ts
process.on("uncaughtException", async (err) => {
  await Sentry.flush(2000);  // drain queue first
  process.exit(1);
});

process.on("unhandledRejection", async (reason) => {
  Sentry.captureException(reason instanceof Error ? reason : new Error(String(reason)));
  await Sentry.flush(2000);
  // Don't exit — unhandled rejection is typically a logical bug
});
```

## 5. Source-map upload

`backend/build.mjs` runs after esbuild and conditionally calls Sentry CLI:

```js
if (SENTRY_AUTH_TOKEN && SENTRY_ORG && SENTRY_PROJECT && release !== "unknown") {
  execSync(`pnpm exec sentry-cli sourcemaps inject ./dist && \
            pnpm exec sentry-cli sourcemaps upload --release="${release}" ./dist`);
  // then strip .map files from dist/
}
```

- Active **only** when all three secrets are present.
- After upload, the `.map` files are deleted from the deploy artefact so
  end users never receive them.
- Sentry retains the maps and resolves stack traces server-side via the
  `release` tag.
- A failure here is logged (`[sentry] source-map upload failed (continuing
  build): …`) but **does not block the deploy** — build artefacts are still
  shipped.

`@sentry/cli` is in `backend/devDependencies` so the binary is available
during the Render build step.

## 6. Required Render env vars

```
SENTRY_DSN              → Render auto-generates this (already set)
SENTRY_AUTH_TOKEN       → operator must paste their Sentry org auth token
SENTRY_ORG              → Sentry org slug (e.g. "subnation")
SENTRY_PROJECT          → Sentry project slug (e.g. "subnation2-backend")
```

`SENTRY_DSN` is already auto-generated by Render. The other three need to
be pasted via the Render dashboard or `update_environment_variables` MCP.
Until they are present, source-map upload is silently skipped and stack
traces show transformed filenames (still readable but not source-line
accurate).

## 7. Verification

After deploy with `SENTRY_AUTH_TOKEN` set:

```bash
# 1. Synthetic 500 on a non-existent route to confirm error capture
curl -i -X POST -H "x-test-token: bogus" \
  https://subnation.ly/api/_test/error-synthetic
# → expect 500 in Sentry dashboard within ~5s, with source-mapped frame

# 2. Confirm correlation_id round-trip
RID=$(uuidgen | tr 'A-Z' 'a-z')
curl -H "x-request-id: $RID" https://subnation.ly/api/healthz -i \
  | grep -i x-request-id
# → response carries the same RID

# 3. Inspect a Sentry event — should show:
#    - tags.correlation_id = $RID
#    - server_name           = srv-d7vv91tckfvc73evnccg-… (Render hostname)
#    - release               = <commit-sha-7>
```

## 8. Owner action

| Item | Status | Action |
|---|---|---|
| `SENTRY_DSN` in Render | ✅ auto-generated | none |
| `SENTRY_AUTH_TOKEN` | ❌ not set | paste in Render dashboard or via MCP |
| `SENTRY_ORG` / `SENTRY_PROJECT` | ❌ not set | paste in Render dashboard |
| Source-map upload | ⏸ skipped until tokens above are set | flips on automatically next build |
| Express error handler | ✅ wired in app.ts | none |
| Global handlers | ✅ wired in instrument.ts | none |
| Sidecar import order | ✅ instrument.ts is first | none |

After provisioning the three Sentry env vars and triggering a redeploy,
backend stack traces in Sentry will be source-line accurate.
